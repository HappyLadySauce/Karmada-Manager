# 产品评估指标框架 (Metrics Framework) - Karmada-Manager 用户体验优化

## 1. 指标框架概述

本框架旨在定义一套清晰、可衡量、可行动的指标体系，用于评估 Karmada-Manager (特别是 Karmada-Dashboard) 用户体验优化项目的效果和成功度。通过持续监测这些指标，产品团队可以了解用户行为、发现潜在问题、验证设计假设，并为后续的产品迭代提供数据驱动的决策支持。

核心目标是提升用户在使用 Dashboard 过程中的**效率**、**易用性**和**满意度**。

## 2. 北极星指标定义

- **北极星指标**: **Karmada-Dashboard 用户任务成功率和满意度**
- **定义**: 
    - **任务成功率**: 用户能够独立、顺畅地通过 Dashboard 完成核心任务（如查看概览获取关键信息、通过表单创建并分发一个应用、修改应用配置等）的比例。
    - **满意度**: 用户对 Dashboard 整体体验的评价，可通过 NPS (净推荐值) 或用户调研问卷中的满意度评分来衡量。
- **选择依据**: 该指标直接反映了用户体验优化的核心价值——用户是否能更容易、更高效地使用 Dashboard 完成其管理和运维工作，并且对使用体验感到满意。它是衡量产品是否真正为用户带来价值的最终标准。
- **衡量方法**:
    - **任务成功率**: 通过关键操作路径的埋点数据分析（如表单提交成功率、特定流程的完成率），结合可用性测试中的任务完成情况。
    - **满意度**: 定期（如每个大版本发布后）进行 NPS调研或用户满意度问卷调查。
- **目标值**: (根据基线数据和版本目标设定，如：任务成功率 > 90%，NPS > 30)。

## 3. HEART / AARRR 等指标体系详述

我们将借鉴 HEART 框架来构建更细致的用户体验评估指标体系。HEART 分别代表：Happiness (愉悦度), Engagement (参与度), Adoption (接受度), Retention (留存度), Task Success (任务成功度)。

### 3.1 Happiness (愉悦度)

- **目标**: 衡量用户对 Dashboard 的主观感受和满意程度。
- **指标**:
    - **NPS (净推荐值)**: "您有多大可能性会将 Karmada-Dashboard 推荐给您的同事或朋友？" (0-10分制)
        - *计算方法*: (推荐者比例 - 贬损者比例) * 100
        - *收集方式*: 定期调研 (如版本发布后1个月内)
    - **用户满意度评分 (CSAT)**: 针对特定功能模块或整体体验进行评分 (如1-5分制，"您对新的概览页面设计满意吗？")。
        - *收集方式*: 产品内嵌调研、用户访谈。
    - **用户反馈中的正面/负面情绪词云分析**: 对用户在社区、GitHub Issues 等渠道的反馈文本进行情绪分析。
        - *收集方式*: 文本分析工具。
- **目标值**: NPS > 30, CSAT (针对核心优化点) > 4.0。

### 3.2 Engagement (参与度)

- **目标**: 衡量用户使用 Dashboard 的深度和频率。
- **指标**:
    - **DAU (日活跃用户数) / MAU (月活跃用户数)**: 每日/每月至少使用一次 Dashboard 的独立用户数。
        - *收集方式*: 后端日志/前端埋点。
    - **平均会话时长**: 用户单次访问 Dashboard 的平均停留时间。
        - *收集方式*: 前端埋点。
    - **核心功能使用频率**: 例如，"表单创建资源"功能每日/每周的使用次数；"概览页"每日/每周的访问次数。
        - *收集方式*: 前端/后端埋点。
    - **页面平均访问深度**: 用户在一次会话中平均访问的页面数量。
        - *收集方式*: 前端埋点。
- **目标值**: DAU/MAU 较优化前提升X%；核心功能使用频率提升Y%。

### 3.3 Adoption (接受度)

- **目标**: 衡量新功能或优化点被用户接受和使用的程度。
- **指标**:
    - **新功能/优化点首次使用率**: 在新版本发布后N天内，体验过某新功能（如"Deployment表单创建"）的用户比例。
        - *收集方式*: 前端埋点。
    - **新功能/优化点渗透率**: 活跃用户中使用过某新功能的比例。
        - *收集方式*: 前端埋点。
    - **"表单创建"与"YAML创建"的使用比例**: 对于支持两种模式的资源创建，用户选择表单方式的比例。
        - *收集方式*: 前端埋点。
- **目标值**: 核心新功能（如表单创建Deployment）的用户接受度 > Z%。

### 3.4 Retention (留存度)

- **目标**: 衡量用户在初次使用后是否持续使用 Dashboard。
- **指标**:
    - **次日/7日/30日用户留存率**: 新用户在首次使用后的第2天/第7天/第30天仍然活跃的比例。
        - *收集方式*: 后端日志/前端埋点。
    - **功能留存**: 使用过某个核心功能的用户，在后续一段时间内再次使用该功能的比例。
- **目标值**: (根据产品特性和用户群体设定，例如：7日留存率 > A%)。

### 3.5 Task Success (任务成功度)

- **目标**: 衡量用户能否高效、顺利地完成其核心任务。
- **指标**:
    - **核心任务完成率**: 用户成功完成预设核心任务（如"通过表单创建一个分发到所有集群的Namespace"）的比例。
        - *收集方式*: 前端埋点（关键步骤成功标志），可用性测试。
    - **任务平均完成时长**: 用户完成核心任务所需的平均时间。
        - *收集方式*: 前端埋点（任务开始和结束事件）。
    - **错误率**: 用户在执行任务过程中遇到错误的频率（如表单校验失败次数、API调用失败次数）。
        - *收集方式*: 前端/后端错误监控。
    - **放弃率**: 用户在任务流程中途放弃的比例。
        - *收集方式*: 前端埋点（特定流程的退出点）。
- **目标值**: 核心任务完成率 > 90%；任务平均完成时长较优化前缩短 B%。

## 4. 功能级评估指标

针对本次优化的重点功能模块，定义更具体的评估指标：

### 4.1 概览页面 (Overview Page)

- **信息获取效率**: 用户在概览页找到特定关键信息（如"异常集群数量"、"Karmada版本"）的平均时长 (通过可用性测试评估)。
- **下钻点击率**: 概览页上可下钻的关键指标（如集群状态统计、资源使用统计）的点击率。
- **页面停留时长**: 用户在概览页的平均停留时间 (过短可能未有效获取信息，过长可能信息组织不佳)。
- **用户反馈**: 针对概览页信息全面性、直观性的用户调研评分或反馈。

### 4.2 资源管理 - GUI 创建/编辑

- **表单创建成功率**: 用户通过表单提交创建各类资源 (Deployment, Service等) 的成功比例。
- **表单创建平均耗时**: 用户从打开表单到成功提交创建的平均时间。
- **YAML 编辑器使用频率**: 在提供表单模式后，用户仍然选择使用 YAML 编辑器的频率 (用于判断表单是否满足大部分需求)。
- **表单字段错误率**: 用户在填写表单时，特定字段校验失败的频率 (帮助定位易错或难理解的字段)。
- **PropagationPolicy 关联成功率**: 用户在创建资源时成功关联或创建 PropagationPolicy 的比例。

## 5. 指标监测计划

- **数据收集工具**:
    - **前端埋点**: 使用合适的库 (如 Segment, Mixpanel SDK, 或自研埋点方案) 捕获用户交互事件。
    - **后端日志**: API 调用日志、错误日志。
    - **APM 工具 (可选)**: 如 SkyWalking, Prometheus + Grafana (部分指标)。
    - **用户调研平台**: 如问卷星、SurveyMonkey，或产品内嵌调研。
- **数据上报与存储**: 确保数据准确、及时地上报到数据仓库或分析平台。
- **数据可视化与看板**: 搭建指标监控看板 (如使用 Grafana, Superset, 或专业 BI 工具)，实时展示核心指标趋势。
- **报告与分析周期**:
    - **每日/每周**: 监控核心KPI（DAU, 任务成功率, 错误率）的波动，及时发现异常。
    - **每双周/每月**: 对比版本发布前后的指标变化，评估优化效果，输出分析报告。
    - **版本回顾会**: 结合数据分析结果，讨论经验教训，指导下一次迭代。
- **责任人**: 明确产品、开发、测试团队在指标定义、埋点、数据分析和报告等环节的职责。
- **持续优化**: 指标体系本身也需要根据产品发展和认知深化进行迭代优化。
