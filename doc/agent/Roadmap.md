# 产品路线图 (Roadmap) - Karmada-Manager 用户体验优化

## 1. 路线图概述

本路线图旨在规划 Karmada-Manager (特别是 Karmada-Dashboard) 用户体验优化项目的迭代计划和关键里程碑。目标是通过分阶段交付，逐步提升 Dashboard 的易用性、功能性和用户满意度，聚焦于概览页面的信息丰富化和资源管理的表单化操作。

## 2. 版本规划策略

- **MVP (Minimum Viable Product) 优先**: 快速交付核心价值，优先实现对用户痛点最明显、价值最高的改进点，即概览页信息增强和核心资源的表单化创建。
- **迭代优化**: 基于用户反馈和数据分析，持续迭代和优化现有功能，并逐步扩展支持更多资源类型和高级功能。
- **与 Karmada 核心版本对齐 (如适用)**: 考虑与 Karmada 主版本的发布节奏进行协调，确保兼容性和新特性的及时支持。
- **分阶段发布**: 将较大的功能模块拆分为更小的版本发布，降低风险，加快反馈循环。

## 3. 详细版本规划

### 3.1 版本 1.0 (MVP - 核心体验提升)

- **主题**: 概览信息增强与核心资源GUI操作
- **目标**: 显著改善 Dashboard 首页的信息密度和直观性；实现最常用资源（Namespace, Deployment, Service, ConfigMap, Secret）的表单化创建和编辑功能。
- **核心功能**:
    - **概览页面 (P0)**:
        - Karmada 控制面状态展示 (AC1.1)
        - 成员集群汇总 (数量、节点) (AC1.2)
        - 成员集群资源汇总 (CPU, Mem, Pod 分配) (AC1.3)
        - 策略信息汇总 (AC1.4)
        - 多云资源信息汇总 (AC1.5)
        - 关键事件展示 (AC1.6)
        - 关键数据下钻链接 (AC1.8)
        - 基础布局与兼容性 (AC1.9)
    - **资源管理 - GUI (P0)**:
        - Namespace 表单 CRUD (AC2.1)
        - Deployment 表单 CRUD (核心字段) (AC2.2)
        - Service 表单 CRUD (核心字段) (AC2.3)
        - ConfigMap/Secret 表单 CRUD (AC2.4)
        - 保留所有资源的 YAML 创建/编辑入口 (AC2.5)
        - 资源创建时关联/创建简易 PropagationPolicy (AC2.6)
        - 资源列表基础功能 (展示、搜索、分页) (AC2.7)
        - 资源详情基础功能 (展示、状态、YAML) (AC2.8)
        - 表单基础校验与提示 (AC2.9)
- **预期交付时间**: [例如：X周后 / YYYY-QZ]

### 3.2 版本 1.1 (易用性与覆盖度增强)

- **主题**: 扩展资源支持与交互优化
- **目标**: 扩展表单化操作支持更多常用资源类型；优化现有表单的交互细节；增强资源详情页信息。
- **核心功能**:
    - **概览页面 (P1)**:
        - (可选) 资源使用趋势图 (AC1.7)
    - **资源管理 - GUI (P1)**:
        - StatefulSet 表单 CRUD
        - DaemonSet 表单 CRUD
        - Job/CronJob 表单 CRUD (简化版)
        - Deployment/StatefulSet 表单增强 (如环境变量、卷挂载、探针的简易配置)
        - 资源详情页增强 (如关联对象列表、更详细状态展示)
    - **可视化资源调度与策略创建 (P1 启动，部分核心功能)**:
        - 集群资源可视化 (AC3.1 - 核心部分：名称、状态、标签、污点、基本资源容量)
        - 图形化配置 `PropagationPolicy.placement.clusterAffinity` (clusterNames, clusterSelector) (AC3.2)
        - 与工作负载创建/编辑流程的基本集成 (AC3.5 - 能够跳转并传递基本策略配置)
    - **非功能性**: 性能初步优化，安全性加固。
- **预期交付时间**: [例如：版本1.0后 M 周]

### 3.3 版本 1.2 (高级功能与体验完善)

- **主题**: 高级策略关联与用户体验细节打磨，可视化调度能力增强
- **目标**: 优化资源与 PropagationPolicy/OverridePolicy 的关联管理；提升整体操作流畅度和用户满意度；完善可视化调度功能。
- **核心功能**:
    - **资源管理 - GUI**:
        - 更完善的 PropagationPolicy/OverridePolicy 在资源创建/编辑流程中的集成与管理。
        - (可选) Ingress 表单 CRUD。
        - (可选) PVC 表单 CRUD。
    - **可视化资源调度与策略创建 (P1 功能完善及 P2 功能)**:
        - 集群资源可视化增强 (如更详细的资源使用情况 - 依赖 AC1.3, AC1.7 的实现)
        - 图形化配置 `PropagationPolicy.placement.spreadConstraints` (AC3.3)
        - 图形化配置 `PropagationPolicy.placement.tolerations` (AC3.4)
        - (可选) 图形化配置 `replicaScheduling` (AC3.7)
        - (可选) 使用预设调度策略模板 (AC3.8)
        - （未来）调度模拟/预览 (初步探索)
    - **调度信息关系树形图 (P2)**:
        - 提供`PropagationPolicy`的调度逻辑的可视化树形图展示 (AC4.1 - AC4.9)
    - **Web Terminal 增强集成**。
    - **用户体验**: 根据用户反馈进行针对性优化，如UI细节调整、错误提示优化等。
    - **可观测性**: 基础埋点数据收集与分析。
- **预期交付时间**: [例如：版本1.1后 N 周]

## 7. 风险管理

- **技术风险**:
    - **前端复杂性**: 表单化创建多种 Kubernetes 资源涉及大量字段和逻辑，可能导致前端代码复杂难以维护。
        - 缓解措施: 良好的组件化设计，加强代码评审，逐步增加资源类型支持。
    - **后端 API 性能**: 概览页数据聚合可能对 Karmada API Server 造成压力。
        - 缓解措施: 优化 API 调用，增加缓存机制，异步加载数据。
    - **兼容性问题**: Karmada 版本升级可能带来 API 变更。
        - 缓解措施: 关注 Karmada 版本发布说明，预留兼容性测试和适配工作量。
- **需求风险**:
    - **用户需求理解偏差**: 对用户真实痛点和使用习惯的理解可能存在偏差。
        - 缓解措施: 加强用户调研，MVP 发布后快速收集反馈，敏捷调整。
    - **需求范围蔓延**: 在迭代过程中不断涌现新需求，导致版本延期。
        - 缓解措施: 严格控制版本范围，重要新需求纳入后续版本规划。
- **资源风险**:
    - **核心人员变动**: 关键技术人员离职可能影响项目进度。
        - 缓解措施: 加强文档建设，知识共享，培养备份人员。
- **依赖风险**:
    - **对 Karmada Metrics Scraper 的依赖 (如需趋势图)**: 若相关组件不稳定或未部署，影响功能实现。
        - 缓解措施: 将依赖功能设为可选或可配置，优雅降级。
