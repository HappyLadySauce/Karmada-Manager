# Karmada Dashboard 前端设计修改规范

## 概述

本文档记录了 Karmada Dashboard 前端界面的设计修改规范，确保界面的一致性、可读性和用户体验。

## 版本信息

- **文档版本**: v1.0.0
- **最后更新**: 2024-05-26
- **适用范围**: Karmada Dashboard React + TypeScript 项目

## 设计原则

### 1. 可读性优先
- 字体大小应确保用户能够清晰阅读
- 最小字体大小不低于 12px
- 重要信息使用 14px 以上字体

### 2. 一致性设计
- 同类组件使用统一的字体规范
- 保持颜色主题和间距的一致性
- 统一的组件样式和交互效果

### 3. 用户体验
- 提供友好的错误处理和空状态展示
- 使用模拟数据确保组件功能演示
- 响应式设计适配不同屏幕尺寸

## 字体规范

### 字体大小标准

| 用途 | 字体大小 | 应用场景 | 示例 |
|------|----------|----------|------|
| 主标题 | 20px-28px | 页面主标题 | 系统概览 |
| 卡片标题 | 16px-18px | 卡片组件标题 | 系统健康状态 |
| 正文内容 | 14px | 主要文本内容 | 描述信息、标签 |
| 辅助信息 | 12px-13px | 次要信息 | 时间戳、状态标签 |
| 最小文本 | 11px | 极小标签 | Tag组件内文字 |

### 字体权重

- **Regular (400)**: 普通文本
- **Medium (500)**: 强调文本
- **SemiBold (600)**: 重要标题
- **Bold (700)**: 主要标题

## 监控组件设计规范

### 系统健康状态卡片 (HealthStatusCard)

#### 字体优化项目

1. **整体状态显示**
   - 标签文字: 14px (原 13px)
   - 状态值: 18px (原 16px)
   - 图标: 18px (原 16px)

2. **系统组件状态**
   - 标题: 14px (原 13px)
   - 组件名称: 13px (原 11px)
   - 状态文字: 12px (原 10px)
   - 图标: 16px (原 14px)

3. **依赖服务状态**
   - 标题: 14px (原 13px)
   - 服务名称: 14px (原 12px)
   - 状态标签: 11px (原 10px)
   - 延迟信息: 12px (原 11px)

#### 布局优化

- 卡片内边距: 16px
- 组件项内边距: 8px-10px (原 6px-8px)
- 图标容器: 32x32px (原 28x28px)
- 最大高度: 240px (可滚动)

### 系统事件告警卡片 (EventAlertCard)

#### 功能特性

1. **数据展示**
   - 最多显示 4 个事件
   - 支持滚动查看更多
   - 空状态友好提示

2. **事件分类**
   - 高危事件: 红色标签
   - 警告事件: 橙色标签
   - 信息事件: 蓝色/绿色标签

3. **时间显示**
   - 智能相对时间: "5分钟前"、"1小时前"
   - 超过24小时显示具体时间

#### 布局规范

- 卡片高度: 320px (固定)
- 事件列表最大高度: 220px (可滚动)
- 事件项间距: 8px
- 边框: 1px solid #f0f0f0

## 模拟数据规范

### 目的
当后端服务不可用或返回空数据时，提供有意义的模拟数据确保组件功能演示。

### 事件模拟数据结构

```typescript
interface MockEvent {
  id: string;                    // 唯一标识
  timestamp: string;             // ISO时间戳
  type: 'Info' | 'Warning' | 'Error'; // 事件类型
  source: string;                // 事件源
  message: string;               // 事件描述
  severity: 'low' | 'medium' | 'high'; // 严重程度
  category: string;              // 分类
  details?: Record<string, any>; // 详细信息
}
```

### 健康状态模拟数据结构

```typescript
interface MockHealthData {
  overall: 'healthy' | 'warning' | 'error'; // 整体状态
  components: ComponentHealth[];              // 组件状态
  dependencies: DependencyHealth[];          // 依赖状态
}
```

## 响应式设计

### 断点规范

| 屏幕尺寸 | 断点 | 栅格列数 |
|----------|------|----------|
| 超小屏 | <576px | 24列 |
| 小屏 | ≥576px | 24列 |
| 中屏 | ≥768px | 24列 |
| 大屏 | ≥992px | 24列 |
| 超大屏 | ≥1200px | 24列 |

### 布局适配

- **概览页面**: xs=24, lg=8 和 xs=24, lg=16 分布
- **监控卡片**: xs=24, lg=12 等分布局
- **集群卡片**: xs=24, sm=12, lg=8, xl=6 响应式网格

- 加载状态: Spin组件遮罩

## 性能优化

### 数据更新策略

1. **概览数据**: 30秒自动刷新
2. **监控数据**: 30秒自动刷新  
3. **事件数据**: 60秒自动刷新
4. **健康状态**: 60秒自动刷新

### 错误处理

1. **网络错误**: 显示友好错误提示和重试按钮
2. **数据为空**: 使用模拟数据或空状态展示
3. **加载失败**: 提供刷新功能

## 最佳实践

### 组件开发

1. **TypeScript支持**: 完整的类型定义
2. **可访问性**: 合适的aria标签和键盘导航
3. **国际化**: 预留i18n接口
4. **测试覆盖**: 单元测试和集成测试

### 代码规范

1. **命名约定**: 使用有意义的组件和变量名
2. **注释完整**: 关键逻辑添加注释
3. **模块化**: 组件职责单一，便于维护
4. **样式管理**: CSS-in-JS和主题变量
